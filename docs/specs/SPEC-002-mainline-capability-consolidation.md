# Hevi 主线能力收敛 · SPEC-002

> 状态:**§3 三项欠账(B1/B2/B3)已全部实施完成,2026-07-13**(见文末 §9 完成记录)。
> 原始状态:草案 v0.1,可交 CC 实施
> 依赖:`Hevi-完整设计-v3.1`(导演台唯一出品引擎 7.1/7.2)、`Hevi-短剧漫剧通道-SPEC-001`(出品形式复用主线的先例)
> 一句话定调(来自产品决策):**主线(longvideo_orchestrator)是唯一能力权威源,必须涵盖并超过 tongjian 的全部通用能力;tongjian 是服务《资治通鉴》的特化出品形式,前端保留,后端通用能力迁移到主线并改为调用,仅保留通鉴专属逻辑自建。三个"不支持"徽章 = 主线能力欠账清单,逐个补齐后摘除。**

---

## 0. 背景:徽章性质的反转

系统扫查发现:tongjian/短剧管线的三项能力(情绪配音、风格参考图、首尾帧关键帧)均在各自最小范围跑通,但**没有一项覆盖主线导演台(DirectorConsole / longvideo_orchestrator)**——主线走 oskill 私有库那套独立的剧本/配音路径,与 tongjian 侧是两套不相干的代码。

**定调前**,三个 DirectorConsole 徽章("不支持")是"两套管线分裂"的诚实探针,应保留。
**定调后**(主线必须涵盖并超过 tongjian),三个徽章反转为**主线的能力欠账清单**,应逐个补齐后摘除。

本 spec 是这次收敛的执行地图。

---

## 1. 目标架构:一个权威源,多个出品形式

```
                能力权威源(唯一)
         ┌──────────────────────────────┐
         │   主线 longvideo_orchestrator │  ← 所有通用能力在此定义与实现
         │   情绪配音 / 风格条件 / 多角色 │
         │   锁脸 / 首尾帧 / …(且更全面) │
         └───────────────┬──────────────┘
                         │ 调用(不自建)
        ┌────────────────┼────────────────┐
    出品形式          出品形式          出品形式
   ┌─────────┐      ┌─────────┐      ┌──────────────┐
   │ 单片向导 │      │ 短剧看板 │      │ tongjian前端  │
   │         │      │         │      │(资治通鉴专属) │
   └─────────┘      └─────────┘      └──────────────┘
                                      特化前置留自建,
                                      通用能力调主线
```

**核心区分(呼应 v3.1 的 7.2 出品形式模型):**
- **能力权威源唯一** = 通用能力的实现只有一处(主线)。
- **出品形式/前端可多** = 通鉴前端、短剧看板、单片向导并存,各有专属交互。
- **不是旁路**:旁路 = 自己实现了一套通用能力;tongjian 转为"调用主线"后不再是旁路,是主线的客户端。

---

## 2. 判定规则:一个能力属于主线还是留 tongjian

**唯一判据:这个能力,别的出品形式(短剧/单片)会不会也要用?**

| 判定 | 归属 | 处置 |
|---|---|---|
| 别的出品形式也会用(通用能力) | **主线** | 迁移到主线实现,tongjian 改为调用,自建版退役 |
| 只有《资治通鉴》场景用(专属逻辑) | **tongjian** | 正当保留自建,是该出品形式的特化前置 |

**示例:**
- 情绪配音、多角色锁脸、风格条件、首尾帧 → 通用,短剧/单片都要 → **主线**
- 针对《资治通鉴》的编年体解析、人物关系图谱、史料特殊编排 → 通鉴专属 → **留 tongjian**

**硬规矩:新能力一律进主线,tongjian 通过调用获得,不在 tongjian 侧自建。** 否则裂缝重生。

---

## 3. 能力欠账清单(主线为"涵盖并超过 tongjian"需补齐)

每项标注改造类型,决定难度与风险。改造类型三档:
- **【接线】** 主线底层已有能力,仅编排层未调用(最易,像 voiceover 多声线那种"能力有没接")
- **【引概念】** 主线私有库(oskill)缺这个概念,需引入新数据模型(最难,触私有库)
- **【新预处理】** 主线缺一道处理步骤,需新增(中,注意敏感区)

| # | 能力(徽章) | 主线现状 | 改造类型 | 敏感区 | 验收门 |
|---|---|---|---|---|---|
| B1 | 情绪配音 | oskill 私有库剧本/配音路径**无"情绪"概念** | 【引概念】 | 私有库数据模型 | 主线多角色对话镜头,逐行情绪驱动 TTS 情感注入,与 tongjian 端到端效果对齐 |
| B2 | 额外风格参考图条件化 | 单集直建走 prompt_style/lighting **文本框**,不走 StylePack;缺"往视频生成塞第二张图"能力 | 【新预处理】 | 生成条件注入 | 主线单集流程可接受一张风格参考图,条件化生成,风格贴合 |
| B3 | 每镜头首尾两张条件图 | 主线无"每镜头指定首尾帧"编排 | 【新预处理】 | i2v 条件注入 | 主线每镜头可分别指定首/尾条件图,过渡自然 |

**顺序建议(按难度递增,先建模式再啃硬的):**
```
先做 B2/B3(新预处理,不触私有库数据模型)→ 建立"主线补能力"的模式
再做 B1(引概念,触 oskill 私有库,最难)→ 有前两个的经验垫底
```
理由:B1 要往私有库引入原本不存在的"情绪"概念,动的是数据模型,最难且回归面最大;B2/B3 是新增预处理步骤,不改私有库核心。先易后难,先建模式。

---

## 4. 各能力改造纪律(触敏感区/私有库的保护措施)

承接工程纪律(guilty until proven innocent / 契约测试锁现状):

### 4.1 通用纪律(所有三项)
- **feature flag 可回退**:每项新能力走开关,能一键回退到主线现状。不做不可逆主干改动。
- **单能力路径零影响**:新能力只在"需要它的镜头"触发(如风格参考图只在提供了参考图时),不改变不使用该能力的镜头的现有路径。回归面从"所有镜头"缩到"使用该能力的镜头"。
- **改前先加契约测试**:动私有库或敏感区之前,先用 golden test 固定现状行为,改完跑测试确认没改坏。

### 4.2 B1(情绪配音,触 oskill 私有库)专项
- 私有库引入"情绪"概念前,先把私有库当前剧本/配音路径的行为用契约测试锁住。
- 情绪字段设计参考 tongjian 已验证的那套(前端 ScriptReviewPanel 的 `l.emotion` 逐行情绪,后端已读通)——**复用已验证的数据结构,不另设计**。
- 优先评估:能否让主线的配音路径**调用 tongjian 已跑通的情绪配音实现**,而非在私有库里重写。若能,B1 从【引概念】降级为【接线】,风险骤降。

### 4.3 B2/B3(新预处理)专项
- 新增的图像条件注入步骤,复用是否可能:tongjian 侧若已有 qwen-image-edit 多图合成能力,评估抽成主线可调的共享件(参考 SPEC-001 的"复用不重造"),而非主线自建第二套。

---

## 5. tongjian 后端收敛处置

tongjian 后端"可接主线、也可保留自建"——给明确边界,防止重新裂开:

| tongjian 现有部分 | 判定 | 处置 |
|---|---|---|
| 情绪配音实现(voiceover.py) | 通用 | 迁主线或抽共享件;迁移期 tongjian 暂留,主线补齐后 tongjian 改调用 |
| 多角色关键帧合成(qwen-image-edit) | 通用 | 抽成主线可调共享件,主线 B2/B3 复用 |
| 多声线合成(vibevoice 多 speaker) | 通用 | 能力已在 tts_service,主线 B1 直接复用 |
| 《资治通鉴》编年体/人物图谱/史料编排 | 通鉴专属 | **正当保留自建**,是 tongjian 出品形式的特化前置 |

**迁移期并存是允许的**(主线补齐前 tongjian 自建版继续跑),但**终态**:通用能力单一实现在主线,tongjian 只保留通鉴专属逻辑 + 调用主线。

## 6. 徽章摘除条件

三个 DirectorConsole 徽章不是永久标记,是待清欠账。摘除条件:

- **B1 完成** → 摘 DirectorConsole.tsx:350 情绪配音徽章
- **B2 完成** → 摘 DirectorConsole.tsx:287 风格参考图徽章
- **B3 完成** → 摘 DirectorConsole.tsx:309 首尾帧徽章

**摘除前每项必须过第 3 节对应验收门**,不达标不摘——徽章诚实是底线,不允许"改了一半就摘徽章"造成 tongjian 那次"以为做完了"的假象。

---

## 7. 明确不做(防过度施工)

- **不合并前端**:通鉴前端、短剧看板、单片向导保留各自专属交互,只收敛后端通用能力。"不是只有一个主线前端"。
- **不在 tongjian 侧加任何新通用能力**:新能力一律进主线。
- **不动通鉴专属逻辑**:编年体/人物图谱等留 tongjian 自建,不强行迁主线。
- **不为清徽章而降级质量**:达不到验收门就不摘徽章。
- **CreativePanel 执行按钮(9项创意工具全未绑事件)是独立坑**,不属本 spec 范围,单独排期;要接则 9 个一起接,不单接首尾帧一个(避免"9个按钮只1个能点"的诡异不一致态)。

**B3(首尾帧关键帧)—— 【新预处理】,未触私有库**
- `orchestrate_longvideo` 新增 `shot_keyframes: dict[int, dict[str,str]]`——纯 hevi 侧字典,不进 oskill 的 `ShotPlan` schema。命中的镜头(索引对上 + 首尾帧文件都存在)整个绕开单图 i2v,改走 `hevi/video/alibaba_maas_service.py::alibaba_maas_keyframe_lock_generate`(`wan2.2-kf2v-flash`,阿里云百炼真实首尾帧生视频 API,复用现有 `ALIBABA_MAAS_*` 凭证)。未命中/文件缺失/生成失败均优雅回退正常单图 i2v,不阻断该镜头。

**B1(情绪配音)—— 复用评估后走【接线】而非【引概念】,未改 oskill schema**
- 评估结论:不需要往 oskill 私有库引入"情绪"概念,也不需要直接调用 tongjian 的实现(架构不兼容——tongjian 逐行调 TTS,主线整片一次调用)。改为在 hevi 侧新增批量情绪推断:`hevi/prompt/emotion_inference.py::infer_line_emotions()`,台词文本产出后另起一次 LLM 调用(全片一次,不逐行)分类,不碰 oskill 的 `ShotPlan`/`Script` 对象。

- `EpisodeRequest.emotion_aware_voiceover` → `DirectorConsole.tsx` 新增复选框(仅 Edge TTS 时可勾)。
