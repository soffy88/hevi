## A. 🔧 状态机纠错(P0,现在的设计是错的)

**v3.0 写的(错误):**
```
draft → ready → generating → verdict → done/rework   ← 把静态事实和动态过程混进一个字段
```

**正确(改成这样):**
```
shot.status  = 只有 pending / ready 两值,只由后端重算,前端只消费
             = 纯"系统流程状态"= 这个镜头的输入齐没齐(静态事实)

"生成中" / "返工中" = 不写入 shot.status
                    = 由 GenerationTask / TaskLink 动态聚合得到(动态过程)
```

**为什么必须改:** 静态事实("输入齐没齐")和动态过程("任务跑没跑")混在一个字段,会导致"一个镜头正在重生成时,它到底 ready 不 ready"这种自相矛盾。

**同时禁止:** 不要把**人工返工、优先级、跳过**等语义混进 `shot.status`。它只回答一个问题:**输入齐了吗?**

### A.1 ready 是"重算"出来的,不是"设置"的

后端按固定规则重算(顺序即优先级):

```
1. skip_extraction = true              → ready
2. 从未提取过                           → pending
3. 提取过但没有任何候选                  → ready
4. 所有资产候选 ∈ {linked, ignored}
   且 所有对白候选 ∈ {accepted, ignored} → ready
5. 其他                                 → pending
```

**铁律:只要还有任意一条候选处于 pending,镜头就不能 ready。**
**例外:某镜头提取后没有任何对白候选,不阻塞 ready**(空镜/无对白镜头正常)。

### A.2 回退必须是自动的

```
原 candidate = linked
→ 用户删除关联 / 替换关联对象 / 清空 scene
→ candidate 自动回退到 pending
→ shot.status 自动重算 → pending
```

必须接入的自动回写点(照抄):
- 提取接口完成后,按镜头同步资产候选 + 对白候选
- 角色/场景/道具/服装关联成功 → 对应候选回写 `linked`
- 删除关联 → 对应候选回退 `pending`
- 同 index 角色被替换 → 被顶掉的旧候选回退 `pending`
- scene_id 从 A 切 B 或清空 → 旧场景候选回退 `pending`
- 接受对白候选 → 写入 ShotDialogLine 并回写 `accepted`
- 忽略对白候选 → 回写 `ignored`

## B. ★ action_beats:镜头是一段动作弧,不是一张画(P0,电影级核心)

**这是本增量最值钱的一条,也是 v3.2 完全没有的。**

### B.1 概念

每个镜头带一组**动作拍点 `action_beats`**,首/关键/尾帧**各抓这段弧的不同阶段**:

| 帧 | 抓哪个阶段 | 含义 |
|---|---|---|
| **首帧** | `trigger` | 触发 / 起始反应 |
| **关键帧** | `peak` | 动作峰值 / 戏剧张力最高阶段 |
| **尾帧** | `aftermath` | 结果态 / 收束阶段 |

**为什么这是"电影级"分水岭:** v3.2 的分镜给的是"这镜头画面是什么"(静态描述);action_beats 给的是**一个动作从触发到峰值到收束的三个采样点**。喂给 i2v 的首尾帧因此天然构成有起承转合的运动,而不是"一张图微微动一下"。**这直接治"人物没有连续的电影一样真实性动作"。**

### B.2 实现:轻量启发式先行,不要求先做结构化对象

```
action_beats: list[str]     ← 就用字符串列表,不做结构化对象
后端按关键词 + 相对位置推断 trigger / peak / aftermath
```

- 首/关键/尾帧**消费同一组 action_beats,但各自抓取不同阶段**
- 阶段信息进 `debug_context`,前端关键帧预览直接展示"动作拍点阶段"和"当前帧优先消费哪个阶段"——**保证编辑页看到的阶段标签与实际生成用的一致**
- **降级:** 镜头尚未确认 `action_beats` 时,继续依赖 `script_excerpt + shot_description` 的自然语言切片(现状行为不变)

## C. ★ 首帧未完成态约束(P0,电影级核心,最反直觉)

**规则:**

> 若剧本存在连续反应链,首帧优先使用**"刚开始 / 尚未完成 / 被打断"的未完成态**表达,**不直接写成后续完成动作或最终姿态**。

**关键词检测(硬约束触发器):**

```
若 script_excerpt + shot_description 中出现连续反应链关键词:
    突然 / 下意识 / 脱手 / 蹲下 / 转身 / 回头 / (可扩充)
→ 追加更硬的时间切片约束
→ 目标:首帧稳定落在"触发后最早的可见瞬间",而不是反应完成态
```

**为什么这条最关键:** 首帧若生成"她已经转过身了"(完成态),i2v 只能让一个已完成的姿态微微晃动 → **视频里没有动作,只有静态图的呼吸感**;首帧若是"她刚开始转身、还没转过去"(未完成态),i2v 才能补出真正的转身运动。

**各帧的专项建议(照抄):**
- **首帧**:优先建立空间、主体初始站位、第一眼视觉印象;存在连续反应链时用未完成态
- **关键帧**:优先锁定动作峰值、情绪爆点或信息密度最高的瞬间
- **尾帧**:优先强调动作收束、情绪余韵、**下一镜头可承接的停留点**

## D. ★ 两层提示词:基础提示词 / 渲染提示词分离(P1)

**当前(v3.2):** 七段式一层,把"这镜头拍什么"和"哪张参考图是谁"揉在一起。

**改成两层:**

```
第一层:基础提示词(base prompt)
  = 只描述画面本身
  ✗ 不含 "图1 / 图2"
  ✗ 不含 "## 图片内容说明"
  ✗ 不承担参考图映射说明

第二层:渲染提示词(frame-render-prompt)
  = f(基础提示词, 当前帧职责 guidance, 当前参考图顺序)
  ← 参考图映射在这一层才注入
```

**意义:** 画面内容(讲故事)和参考图映射(讲工程)是两个关注点,混在一起两边都写不好。分层后基础提示词可复用、可跨帧共享。

## E. ★ director_command_summary:约束按风险动态排序(P1)

**不是简单拼接所有 guidance,而是按帧类型重新排序,并按风险动态提级:**

```
若镜头存在:同场景连续关系 / 对白 / 明显对视风险
→ screen_direction_guidance 自动上提
→ 更早进入「必须」级约束(而非「优先」级)

因此:首帧 / 关键帧 / 尾帧 看到的「必须 / 优先」摘要并不相同
```

**目标:让模型先守住最容易导致突兀感的事项**——
- 同场景镜头的左右轴线
- 对话关系中的视线连续
- 推进镜头中的构图重心

**意义:** prompt 不是一堆并列约束,是**有优先级、且优先级随镜头风险动态变化**的。这比静态模板高一档。

## F. ★ prompt 编译规范(P1,四条)

### F.1 统一输出口径(降低镜头间漂移)

基础提示词固定按此顺序组织:

```
1. 景别 / 机位 / 运镜
2. 场景环境
3. 主体人物或关键对象
4. 动作与状态
5. 氛围与情绪
6. 必要的风格收束
```

同项目所有镜头同一结构 → 漂移自然小。

### F.2 主体优先级:禁止平均铺陈

```
优先写主角色
优先建立主场景
道具 → 仅在进入主动作或构图焦点时才重点写入
服装 → 主要用于保证人物外观一致性,不要求每次成为画面中心
```

治"什么都写一点,结果画面没重点"。

### F.3 已确认实体名称必须原样保留(★ 直接治"乱跳")

```
✗ 不应被改写 / 翻译 / 替换为同义词
✓ 镜头信息不完整时,优先用已确认实体补足画面
✗ 禁止凭空新增主要主体
```

**这是"禁止模型自己发明新角色/新场景"的硬规则。**

### F.4 项目级风格是显式约束,不是弱提示

```
必须继承 项目级 visual_style
必须继承 项目级 style
若 unify_style = true:
  → 镜头信息不足时,优先向项目既有风格收敛
  → 不应自由发散到其他视觉风格

链路:项目风格 → 基础提示词 → 渲染提示词 → 图片生成(连续,不断链)
```

## G. ★ 候选表机制:AI 候选 vs 确认实体分离(P1)

**这是 SPEC-003 ③设计清单 / ④台词行该用的机制:AI 提的是候选,人确认后才成为事实。**

### G.1 资产候选表 `shot_extracted_candidates`

```
shot_id
candidate_type      : character / scene / prop / costume
candidate_name
candidate_status    : pending → linked | ignored
linked_entity_id
source / payload / confirmed_at
```
生命周期:`pending → 用户确认 → linked 或 ignored`

### G.2 对白候选表 `shot_extracted_dialogue_candidates`

```
shot_id
index
text
line_mode
speaker_name
target_name         ← ★ 见 H
candidate_status    : pending → accepted | ignored
linked_dialog_line_id
source / payload / confirmed_at
```
生命周期:`pending → 用户接受 → 写入 ShotDialogLine 并置 accepted;或 ignored`

**关键:候选表记录的是"确认明细",不是最终资产本身。** 最终资产是 Subject/ShotDialogLine。

## H. ★ target_name:对谁说(P1,eyeline 的数据来源)

**v3.2 的对白台词行只有 `speaker`,漏了 `target`。**

```
对白 = A 对 B 说话
speaker_name = A(谁说)
target_name  = B(对谁说)  ← 新增
```

**为什么重要:target 直接决定 eyeline** —— A 对 B 说话,A 就该看着 B。v3.2 刚加的 `eyeline` 维度**不用另外标注,对白的 target 就是它的数据源**。

## I. ★ skip_extraction:逃生阀(P1,不加会卡死)

```
skip_extraction: bool   ← 用户明确声明"当前镜头无需提取"
= true → shot.status 直接 ready
```

**没有这个阀门,空镜 / 转场 / 无角色镜头会永远卡在 pending。** v3.2 的 readiness gate 缺这个,会卡死。

## J. ★ 相邻镜头上下文(P2)

基础提示词生成时带入:

```
上一镜头:标题 + 剧本摘录 + 结尾状态摘要
下一镜头:标题 + 剧本摘录 + 起始目标摘要
当前镜头与相邻镜头的连续性建议
```

追加的连续性约束:
- 存在上一镜头 → 当前画面优先**承接上一镜头的动作、视线、情绪或空间方向**
- 存在下一镜头 → 当前画面形成**可自然过渡到下一镜头的收束状态**
- 尽量明确主体位置、朝向与动作峰值,减少镜头切换突兀感

追加的构图与空间锚点建议:
- 按景别与运镜判断优先突出**主体**还是**环境**
- 同场景连续镜头优先保持**空间轴线与人物朝向稳定**
- 存在主场景和主角色时,**显式提示模型锁定场景锚点与主体视线逻辑**

追加的朝向与视线建议:
- 对话/对视场景 → 保持**说话者与受话者的视线关系连续**
- 同场景相邻镜头 → 保持**人物左右面向稳定,减少反打跳轴**
- 过肩镜头 → 保持**前景肩部位置与被看对象方向稳定**

> **与 v3.2 观察态机制(4.0.1b)的关系:** J 是**计划态**双向连续性(上下镜互相照应),观察态是**事实态**单向修正(实际末帧覆盖计划起始态)。**两者叠加:先按 J 编译计划,再按观察态用实际末帧覆盖起始状态。** 不冲突,是互补。

## K. ★ 可观察性:debug_context + quality_checks(P2)

**这是 prompt 层的 decision_trail(呼应 v3.x 的 verdict 数据资产化)。**

基础提示词任务成功后,除返回最终 `prompt`,还附带:

```
debug_context:
  - 本次实际使用的 项目风格字段
  - 本次实际使用的 镜头语言字段
  - 本次实际使用的 已确认实体上下文
  - action_beats 推断出的阶段 + 当前帧优先消费的阶段

quality_checks:
  - 各项检查结果
```

用途:排查**"为什么生成成这样"**。前端关键帧预览直接展示阶段信息,保证编辑页所见 = 实际所用。

### K.1 轻量质量闸 + 自动修正重试

```
若结果混入 "图1 / 图2" 或 "## 图片内容说明"  → 自动带修正要求重试一次
若结果遗漏主角色名称                          → 自动带修正要求重试一次
再决定最终落库
```

## L. ★ 聚合状态协议 + 页面职责边界(P2)

### L.1 聚合状态:前端不自己拼状态

```
GET  /shots/{id}/preparation-state
返回聚合:
  shot
  assets_overview
  dialogue_candidates
  saved_dialogue_lines
  pending_confirm_count
  ready_for_generation
```

**所有 mutation 接口统一返回最新聚合状态**(`{action, state}`),前端**不再手写** `loadAssetsOverview()` / `refreshCurrentShot()` / 自己拼 `pendingConfirmCount`。

**前端硬约束:不再自行推导正式 `shot.status`,不再手动把 pending/ready 写回本地状态。**

### L.2 页面职责边界:准备 vs 生成 严格分离

```
准备页(单镜头准备台)
  职责:提取资产/对白候选 → 确认候选 → 把镜头推进到 ready
  凡是影响 skip_extraction / 候选状态 / shot.status 的动作,都在这一页

生成台(章节内生成工作台)
  职责:查看 video-readiness → 生成关键帧/参考图 → 调视频参数 → 生成 → 批量推进
  它关注:能不能生成、还差哪些前置、先关键帧还是直接视频

主流程:准备页 → 提取与确认 → ready → 生成台 → readiness → 关键帧/参数 → 生成
```

**生成台里的"信息确认状态"只作为诊断入口,不承担主要确认动作。**
