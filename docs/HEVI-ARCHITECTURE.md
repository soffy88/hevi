## 0. 前置判断:一个技术趋势分界线

所有设计取舍都基于这一个押注:**未来 12–18 个月,视频生成能力会持续商品化,编排/资产/裁决/路由不会。**

### 会被模型层吃掉的(不自建,租)

- 单镜头画质、物理合理性 —— 每季度换代,自建即贬值
- lip-sync —— Veo3 已原生音画,Kling/海螺在跟进
- 运镜控制 —— 正从"专门功能"退化为"prompt 里一句话"
- 特效(Pikaffects 类)—— 纯模型能力

### 模型层吃不掉的(这是 Hevi 该建的)

- 跨镜头、跨集的叙事与身份一致性
- 成片质量的裁决与定向返工
- 成本感知的多 provider 路由
- 人在环上的编排

**设计总纲:凡是生成能力,一律外采并可热替换;Hevi 的本体是编排、资产、裁决、路由四件事。**

## 1. 定位与北极星

**定位一句话:** Hevi 不是"AI 视频创作工具"(那是 Runway),是**系列化视频的自动化制片厂**——输入剧情,输出"同一个 IP 的第 N 集",质量有下限保证,成本有上限保证。

**北极星指标:** 一个用户能否在不碰任何参数的情况下,稳定产出"同一个角色、同一风格的第 47 集",且单集废片率 < 5%。

**为什么是这个:** 单片生成工具全球至少 50 家,拼模型接入速度没有护城河;"稳定量产系列"要求角色一致性 × 风格固化 × 流程编排 × 质量闭环四者同时成立——这是唯一能支撑订阅制(按月产能)而非积分零售的产品形态,也是竞品补不了的差距。

## 1.5 护城河重新标定(v2.1 关键修订)

调研 SamurAIGPT/Generative-Media-Skills 及其商业闭环(muapi.ai 聚合器 → muapi-cli 原语 → 41 个工作流配方 → Open-AI-UGC 成品 SaaS)后,必须修正一个原设计里的判断误区。

**误区:** 原设计把"多 provider 接入 + 能力矩阵"列为四大护城河之一。

**现实:** 多 provider 接入层已被彻底商品化。muapi.ai 一类聚合器已把 100+ 模型(Seedance/Kling/Veo3/Flux…)聚合成一行 CLI 调用,加一个新模型约 10 行代码。**"接了很多 provider"不再是任何人的护城河。**

**重新标定后,Hevi 真正的护城河只有三样,全部在接入层之上:**

1. **路由治理**——不是"能调用哪些 provider",而是"用 verdict 历史质量数据决定每个镜头该走谁"。聚合器给的是原始接入能力,它永远不会告诉你"这个 provider 在国风水墨特写上的历史 identity_distance 是多少"。这把原本 L0 的 v2(数据驱动路由,见 3.2)和 5.3.4 的自我校正闭环,从"锦上添花"提升为**唯一真正的接入层护城河**。
2. **verdict 数据资产**——竞品的 review 步骤是浅层的(让 LLM 看一眼就过),没有 shot_verdict 持久化 + 身份 embedding + 分层校验。别人有"审查"这个动作,但没有把审查变成累积数据资产(见第 6 节)。
3. **Series 一致性继承**——单片赛道已被商品化到极致(SamurAIGPT 名下 69 个单片 SaaS 模板即活体证明),唯一没被商品化的是"同一个 IP 稳定量产第 N 集"的字段继承机制(见 5.4/5.6)。

**战术推论:** 既然聚合器已存在,可将其作为 L0 能力矩阵里的**一个兜底供给条目**(见 3.4)。当自建的直连 provider 余额耗尽时(如当前 fal/DashScope 双欠费),多一条兜底路径,而非自己逐个维护每个上游接入。把竞品的基础设施为我所用,价值继续留在其上的治理层。

**战略纪律:** 不要被 SamurAIGPT 的广度诱惑。它做 100 个浅产品,Hevi 做 1 个深产品。看到它 69 个仓库的正确反应不是"我也该多做几个品类",而是"这就是不砸深度的下场——沦为可被一行 CLI 替代的壳"。v2 里砍品类、砸深度的判断,被这个竞品反向验证为正确。

## 2. 五层架构总览

```
                    ┌─ 出品形式(导演台的输出档位,同一引擎按复杂度变薄变厚)─┐
                    │  单片 · Series · 头像解说 · 短剧/漫剧 · …               │
                    └───────────────────────┬───────────────────────────────┘
L4 导演层    Producer / Director / Editor —— 唯一出品引擎,Workflow-as-MCP 编排范式
L3 裁决层    hevi-verdict —— 遍历「一致性挂载树」逐节点校验 + 分层校验 + 结果持久化
L2 资产层    资产供应链(矿工采集→检索→锁定) → 五类资产实体(角色/场景/道具/服装/声线,挂载成树)
             Subject(2D身份资产 + 3D视角资产,互补共存)/ Series / StylePack / Template
L1 执行层    镜头生成分层流水线(身份→场景→动作→语音→口型,每层独立选模型)+ 装配 + lint
             ┌ readiness 状态机:备齐才开机(draft→ready→generating→verdict→done/rework)
L0 供给层    Provider 能力矩阵(含 ComfyUI 本地统一入口 + 口型专用层)+ 分层成本路由 + 预算熔断
```

**v3.0 一句话:** v2.x 确立了引擎的骨架(五层 + 导演台唯一出品),v3.0 填的是让它到顶级的密度——**生成拆成分层流水线、资产扩成五类挂载树、一致性从锁向量升级为遍历树、生产从抽卡升级为 readiness 工序、锚点从生成升级为"检索优先生成兜底"。**

### 2.1 导演台是唯一出品引擎(v2.4 确立)

**所有成片都是导演台的"出品形式",不设任何旁路。** 单片、Series、头像解说、短剧/漫剧,是同一条管线在不同复杂度下的自然形态,不是各自独立的管线。

- **薄厚自动决定,不是两条路**:单片进来,故事解析/剧集规划等特化前置层空转,只走"一个 Series、一集、若干镜头"的最小路径;短剧进来,同样的引擎点亮那些层。**"不是简单的走旁路,是复杂的点亮更多层。"**
- **为什么不给简单出品开旁路**:旁路 = 第二套生成/质检/路由 = 又一个自立门户的入口。它会慢慢长出自己的一致性逻辑并与主管线漂移,几个月后又需要一次"短剧管道诊断改造"式的收口。短剧已经演示过这个故障模式,不给单片再留一个。
- **收益**:单片也享受与短剧完全一致的身份一致性/verdict/成本路由(单片同样需要,凭什么旁路就不要);而只维护一套引擎。
- **和上一层修正的呼应**:短剧之所以能"不自立门户、只做出品形式级差异",前提正是导演台强到内建了「Workflow-as-MCP 编排 + ComfyUI 本地供给 + 身份锚点 + verdict」。**加强导演台,是让一切特化出品都不自立门户的釜底抽薪。**

依赖关系是硬的:**没有 L3 裁决(尤其是持久化数据),L0 的路由无法从"猜"升级成"算";没有 L0 路由治理,L4 的 Agent 只是更快地烧钱产烂片;没有 L4 的计划级反思,自动化程度越高,烧穿预算的风险越大。** 建设顺序不能颠倒。

## 3. L0 供给层:从 fallback 链到交易所

现状问题:provider 是硬编码优先级链,系统对余额/配额是盲的,只能靠 403 感知欠费。

### 3.1 能力矩阵

一张数据表,每个 provider 一行,列为:`{t2v, i2v, 参考图条件, 原生音频, lip-sync, 最大时长, 分辨率档, 单秒成本, 当前健康度, 剩余余额, last_verified}`。

- **`last_verified` 字段**:路由决策前检查是否过期,过期条目降权或跳过——provider 会悄悄换模型版本、降配额,矩阵不能一直被信任到 403 才发现问题。**实例(v2.3):** 外部情报显示 Sora 2 已停运——若矩阵曾收录 Sora,这正是 last_verified 该触发下线的典型情形;model ID 停运/换代是常态,新鲜度字段就是防这个。
- **`stock_footage` 作为一种零成本供给类型**:严格限定在无主体绑定的镜头(scene 类空镜/转场/B-roll),不允许用于任何带 character/portrait/product 参考图约束的镜头。素材需选用有清晰商用授权的库(如 Pexels)。**对位校验(v2.3)**:素材命中不能只靠检索,还要过一道对位分——`match_score(caption, b-roll)`,低于阈值(如 <0.3)判为"未对上",触发换素材或回退生成。keyword_map 必须**按 StylePack/内容领域定制**,不能用通用示例词典(中性占位词不会 match,反而干扰对位)。这补上了"素材检索到了但对不对得上这句旁白/这个镜头"的校验缺口。

**实现范式(可借鉴 SamurAIGPT):** 能力矩阵应实现为一个中心化的 **schema 文件驱动**(参考其 `schema_data.json` 做法)——所有 provider 的能力/端点/参数/定价由单一 schema 描述,路由与适配代码从 schema 读取而非硬编码。这直接兑现"加新 provider = 加一行数据,不改路由逻辑"的设计目标,是能力矩阵存储格式的现成实现参考,不必自行设计。

### 3.2 成本感知路由

- **v1(镜头粒度)**:每个镜头请求携带 `(质量要求, 预算余量, 时限)`,路由器在健康 provider 中解 `min(cost) s.t. capability ⊇ 需求`。同一条片,主角特写走支持参考图的高写实云,空镜走本地零成本模型或 stock_footage。
- **v2(依赖 L3 数据积累)**:从静态能力布尔值升级为经验质量分布——例如"Kling 在角色特写 + 国风水墨 StylePack 组合下,identity_distance 历史中位数是多少",路由从声明式约束变成数据驱动的期望值优化。

### 3.3 余额探针 + 三层预算熔断

- 定时查各 provider 余额/配额,查不到的用滚动 403 率做代理指标,接入 Aegis 告警。
- 三层熔断:全局日预算 → provider 日预算 → 单任务预算。
- **质量维度告警**:provider 的 identity_distance / style_distance 滚动均值出现台阶式跳变,视为"质量维度的余额耗尽",走同一条告警通道。

### 3.4 聚合器作为兜底供给条目(v2.1 新增)

将 muapi.ai 一类多模型聚合器,作为能力矩阵里的**一个特殊 provider 条目**接入——它本身聚合了 100+ 模型,单点接入成本极低。

- **定位:兜底,不是主力。** 直连 provider(fal 直连、本地 GPU)成本更透明、可控性更强,是主力;聚合器作为余额耗尽或直连全线故障时的兜底供给路径。当前 fal/DashScope 双欠费导致全线 403 的情况,若有聚合器兜底条目,可避免"零可用供给"的硬停机。
- **代价与边界:** 聚合器多一层加价、可控性下降、且把一部分供给依赖交给第三方。因此它在路由优先级中排在直连之后,仅当成本感知路由在直连集合内无解时才启用,并计入成本估算。
- **战略意义:** 把竞品的基础设施层为我所用,Hevi 的价值继续留在其上的治理层(路由/verdict/一致性)——这三样聚合器都不提供。

### 3.5 ComfyUI 作为本地供给统一入口(v2.4 新增)

本地供给(wan_local / ltx2_local 等)此前是逐个裸接模型,接线零散。改为**统一走 ComfyUI**作为本地供给层的单一入口(参考 Pixelle-Video 的双执行模式:本地 ComfyUI + 云端 ComfyUI)。

- **收益:复用 ComfyUI 生态现成节点。** i2v、lip-sync、ControlNet 身份锁定、IP-Adapter 等工作流在 ComfyUI 生态里都是现成的,不必自己逐个封装。换模型 = 换 workflow 节点,不改管线代码——这与能力矩阵"加新 provider = 加一行"的目标同构,只是落到本地供给的具体实现。
- **服务对象:所有出品形式的 economy 档**,不是短剧专属。单片、Series、短剧的本地零成本档统一从这里出。这正是"加强供给层、让一切出品继承"的体现——不在任何单一出品形式里另接本地供给。
- **与身份铸模成本策略咬合(见短剧改造附录)**:"贵模型造身份图 + 本地免费 i2v 补动作"里的"本地 i2v",实现载体就是 ComfyUI 的 i2v workflow,身份锁定节点现成可用。
- **与 Workflow-as-MCP 呼应(见 7.x)**:ComfyUI workflow 天然可被包装为 MCP 工具,与导演台的编排范式无缝衔接。

## 4. L1 执行层:镜头生成分层流水线 + readiness 工序

**v3.0 核心重构:一个镜头的生成,不再是"一次调用",而是一条串联的分层流水线。** 这是 v3.0 把 Hevi 推到顶级的最关键一处——之前的"整镜头一次性生成",把不同时间尺度的一致性约束(口型的毫秒、动作的镜头内、场景的整集、身份的跨集)全压给单次生成,导致口型/场景/动作三类问题同时出现。分层后,每类约束在各自正确的环节、以各自的锚点介入。

### 4.0 镜头生成分层流水线(v3.0 新增,L1 的新主干)

一个镜头 = 在指定机位下,投影锚点 + 补动作 + 贴音画,五层串联:

```
镜头生成流水线(每层可独立选模型):
  [身份层] 从角色资产按机位渲身份帧(四视图/Subject3D)┐
  [场景层] 从场景资产按同一机位渲背景帧               ┼→ [动作层] i2v 补动态(按动作复杂度选档)
                                                       ┘         ↓
  [语音层] TTS 生成对白/旁白(带情绪标签)──────────────→ [口型层] 专用模型对齐口型
                                                                 ↓
                                            装配(多轨对齐 + 交付门质检)
```

**关键:机位是贯穿身份层和场景层的查询参数,不是单独一层。** 同一场戏的正反打两个机位,身份帧和场景帧来自同一对资产的两次不同视角投影,所以两个机位天然是"同一个人在同一个空间"——正反打不穿帮的根因在此(详见 5.7 机位驱动渲染)。

#### 4.0.1 分层模型路由表(接 L0 成本路由,回应"贵模型用不起")

**精细模型只出现在"铸模"环节且一次性;海量的逐镜头生成走便宜档。**

| 层 | 锚点/职责 | 模型档 | 频次 |
|---|---|---|---|
| 身份层 | 角色资产铸模(四视图/3D) | 精细档(Veo/Kling),一次性 | 极少 |
| 场景层 | 场景资产铸模 | 精细档,一次性/场景 | 少 |
| 动作层-简单 | 有身份+场景锚点,i2v 补动态 | 本地/便宜云(ComfyUI i2v) | 海量 |
| 动作层-复杂 | 打斗/施法/多人 | 中高档(Kling 物理合理性) | 少量 |
| 语音层 | TTS 生成对白/旁白 | 合适 TTS(声线档案,见音频子系统) | 每句 |
| 口型层 | 画面+音频对齐 | 专用音画同步模型 / Duix,不指望动作层附带 | 对白镜头 |

**成本本质:一次性用精细模型把身份+场景两个锚点铸死,便宜模型在锚定的身份+场景里演动作,专用模型把口型贴上去。** 这就是"用旗舰模型的脸,跑本地模型的价"。注意反直觉点:**动作层"简单用便宜、复杂必须上档"**——复杂动作的时序连贯和物理合理正是便宜/本地模型最弱处,省不得。

### 4.0.2 readiness 状态机:备齐才开机(v3.0 新增)

**把"准备好"和"正在生成"严格分离**,每个镜头有统一的 readiness 状态决定它能否进入生成队列:

```
镜头状态机:draft → ready → generating → verdict → done / rework
                    ↑
              readiness gate(硬闸门)
```

`ready` 的判定 = 该镜头的所有输入齐备:身份锚点在、场景锚点在、StylePack 锁定、机位已定、prompt 已过 lint(4.2)、(对白镜头)音频与情绪标签已备。**任何一项缺失 = not-ready = 禁止进生成队列。**

**为什么这是顶级和能用的分水岭:** 烂片很多时候不是生成质量差,是**输入没齐就被送进了生成**(短剧自立门户出烂片的一个直接原因)。readiness gate 把生产从"抽卡碰运气"变成"备齐才开机的工序",这是工业级流水线与作坊的区别。

### 4.1 三个手术(沿用)

现有管线(逐镜头 + 双变体 + checkpoint + 装配)健康,分层流水线是在其内部结构化,不推翻。

### 4.1.1 三个手术

1. **i2v 前端暴露**——后端已接线,补前端入口。
2. **lip-sync 作为可路由能力**——即口型层,能力矩阵加一列,路由到专用音画同步模型,不作为独立功能开发。
3. **镜头级返工接口**——`regenerate(task_id, shot_ids[], hints)`,复用已有 checkpoint;接 readiness 状态机的 rework 态。

### 4.2 生成前 lint(Stage A,发生在花钱之前)

Prompt 富化产出后,跑一遍确定性字段级自检(依托 4.3 的七段式结构化 schema,逐字段查):

- 角色段:是否混入抽象形容词(禁用"时尚的""好看的"等,只允许可视化特征)
- 角色段:结尾是否带身份锁定句("全片保持一致的身份、服装、发型、外貌")
- 摄像段:是否已调用某个设备包
- 时间轴段:每个镜头是否塞入一个非完美事件,且全片不重复(仅 verité 类 StylePack 启用,见 6.3)
- 全局:负面约束块是否齐全、抽象词是否已翻译为物理现象
- Hevi 特有:IP 安全改写是否已过(见 5.5)、时长字段是否与目标时长档匹配

零生成成本,拦住"这条 prompt 大概率生成出来就不合格"的问题。这套 lint 的检查项直接源自 seedance 方法论的"输出前八项自检",是成熟可抄的现成清单。

### 4.3 返工接口升级

`regenerate(shot_id, diagnosis_category, hint)`,诊断分类取自固定分类表:

`运镜 / 光照 / 动作 / 参考图角色错配 / 时长 / 构图 / 音频 / 安全词误触发`

让重新富化 prompt 的一步知道该拉哪根杠杆,而不是盲目重掷。**重试次数硬上限(建议 3 次)**,超限走降级交付 + 部分退款,不无限烧预算。

### 4.4 装配阶段新增三项

- **跨 provider 调色统一**:按 StylePack 的调色基准做一次 ffmpeg 归一化,解决多 provider 混剪接缝明显的问题。
- **转场对齐 BGM 节拍点**:节拍提取 + 转场卡点,零边际成本的观感提升。
- **多轨同步对齐范式(v2.3 新增,根治过压缩类 bug)**:音/画/字幕三轨的裁剪必须由**一份统一 cut list 驱动**,而非各轨独立处理。三条实战踩坑纪律:
  - 音频裁剪用 `atrim + concat`,**不要用 `aselect`**(aselect 对音频常剪不干净,是过压缩/音画错位的常见根因)
  - 视频裁剪用 `select + setpts`
  - 字幕时间戳走 `remap_time(t, cut_list)` 重映射,与音画保持同一时间基准
  - 素材进装配前先 `batch_normalize` 对齐统一 fps(默认 30)+ 去背景音,防止"素材 fps 与 timeline 不符导致播放卡格/变速"。这是 v2.2"14 镜头压成 6s"过压缩 bug 的范式级根治——问题本质是多轨时间基准不统一,统一 cut list 从源头消除。

### 4.5 长视频→竖屏切片分发(可借鉴,外采,非自建)

内容创作者的高频刚需:一条长视频→多条竖屏短片分发。SamurAIGPT 的 AI Clipping 已把这件事完全服务化——服务端转写 + 病毒性排序 + 去重 + 人脸追踪自动裁剪,一次托管 API 调用完成,无需本地 Whisper/LLM。

- **原则:外采,不自建。** 与总纲"纯生成/纯处理能力全部外采"一致——连剪辑都在服务化,再次验证不该自建。
- **落位:** 作为 Series 成片的一个下游分发导出选项,而非核心链路。Series 出一集长视频后,可选"切片分发"→ 调用外部切片服务 → 得到多条竖屏短片。Hevi 的价值在"稳定产出可切片的高质量长视频",切片本身外包。

## 5. L2 资产层:一致性不是功能,是资产的属性

这是与所有竞品拉开身位的一层。**v3.0 把这一层从"锁两个向量"升级为一套完整的资产体系:上游有供应链(锚点从哪来)、中层有五类实体挂载成树(锁什么)、下游有多视图/3D 按机位渲染(怎么用)。**

### 5.0.1 资产供应链:锚点"检索优先,生成兜底"(v3.0 新增,L2 上游)

不是每个锚点都要贵模型从零生成——大部分常见形象/场景,网上现成素材就够接近,检索+微调即可。给资产层加一条上游供应链:

```
资产矿工(后台持续采集,能挖就挖)
  → 源资产池(只存"参考",每个资产带检索向量)
      ↓
导演台读剧本设定 → (过 StylePack 氛围词典把抽象描述译成具象特征)→ 编码成检索向量
      ↓
  池中检索最接近的候选
      ├─ 相似度达标 → 当参考图喂生成模型 → 模型重新生成 → 修正锁定为本剧锚点
      └─ 相似度 < 阈值 → 回退到贵模型从零铸模(复用 stock_footage 的 match<0.3 回退机制)
      ↓
  锁定的锚点 → 进五类资产实体(5.0.2),生成一致性向量
```

**两个向量,别混:**
- **向量甲(检索向量)**:资产的视觉内容 embedding(CLIP/DINO),用途是从海量池子里**找**候选,要召回广。池中每个资产都有。
- **向量乙(一致性向量)**:被选中锁定后的锚点 embedding,用途是生成后**校验**漂移(verdict 的基准),要身份精确。只有锁定的锚点才有。

**版权:采集端不做分区(能挖就挖),因为消费端守死一条架构约束——源资产只当"参考"influence 生成,一个像素都不进成片,最终画面是模型重新生成的。** 参考层与成片输出层物理不连通。用户输入侧的 IP 过滤(5.5)照旧保留,那是另一回事(守用户上传的明星脸/IP,与资产池无关)。

**资产矿工是独立后台服务**(挂 Aegis 管理),慢速持续跑,与实时出片解耦。绿色可商用素材现在就能开始攒——和 verdict 数据一样越早攒越值钱、无法补录。

### 5.0.2 五类资产实体 + 挂载树:一致性是"锁一棵树",不是"锁一个向量"(v3.0 核心升级)

真实短剧工业现场的一致性,不是锁脸,是把**角色/场景/道具/服装/声线全部做成跨镜头共享实体**,挂载成树:

```
角色(character:身份)
 ├─ 服装(wardrobe:可切换,每套锁定)      ← 治"女主从头到尾一套衣服"/换装后不一致
 ├─ 发型(hairstyle:可切换,每款锁定)
 └─ 声线(voice:锁定)                      ← 身份一致从"脸一致"扩到"脸+声一致"
场景(scene)
 ├─ 光照(lighting:时段锁定)
 └─ 道具(prop:锁定)                       ← 反复出现的关键信物/武器每次一致
镜头一致性 = 遍历这棵树,每个节点独立锁定、独立校验、可独立切换
```

**这是"顶级"和"能用"的分水岭:** 能用的锁脸;顶级的锁脸+衣+发+声+道具+光,每一类独立成实体。v2.x 只锁了身份(character)和风格(StylePack),v3.0 补齐服装/道具/声线三类,并确立**分层挂载**——服装/发型/声线挂角色,光照/道具挂场景。

**verdict 因此升级(见 6.x):** 从"算 identity_distance + style_distance 两个数",升级为"**遍历这棵资产树,逐节点校验每一类资产的一致性**"。观众吐槽的"一套衣服打巴掌关门动作有错"这类穿帮,根因就是这些子节点没被当独立实体管理。

### 5.1 Subject:多视图 + identity embedding

两条设计原则:

1. **姿态不变**:只编码"是谁",不编码"在做什么"。身份和运动是两个独立的向量空间,不能混合表示,否则既做不好一致性校验,也做不好动作驱动。
2. **多区域、遮挡鲁棒**:脸部 embedding(ArcFace/InsightFace 一类)+ 发型/服装区域 embedding(裁剪躯干过 CLIP/DINO)+ 体型轮廓向量分开算。verdict 校验时按当前帧实际可见区域动态选用哪个向量比对,侧脸、背影镜头也有校验依据。

**边界:** 这套向量只用于生成后的一致性校验,不用于驱动生成——云端 provider 是黑盒,喂不进 latent space。唯一能做"向量驱动运动"的地方是 **Duix 本地数字人**(关键点驱动的姿态/手势迁移),因为那条线拥有完整控制权,值得单独投入。

**共存说明(v3.1):** identity embedding(2D 身份资产)和 Subject3D(3D 视角资产)**不是低保真/高保真的替代档,是互补共存的两种资产**——2D 强在身份保真、3D 强在视角自由。同一角色同时拥有两者,镜头级按需组合(见 5.7)。实测数据(2D 身份分 0.77-0.84 > 3D 0.61)证明 2D 是当前更可靠的身份来源,3D 是唯一的多视角来源,该合用不该二选一。

#### 5.1.1 四视图标准包(v3.0 新增,2D 多机位的最低标准)

2D 身份档不是"随便几张图",是一个**结构化的四视图标准包:正 / 侧 / 背 / 面部特写**(参考天工的多视图数字资产做法)。原因是多机位:

- 正反打对话 → 需要侧视图
- 越轴到背后 → 需要背视图
- 大脸特写 → 需要特写视图
- 只有一张正面图时,模型生成侧/背是在**猜**,每次猜的不一样 → 这是"人物不连续"的一个深层来源

**四视图是 Subject3D 的前置和降级档:** 3D 未就绪时,四视图撑住大部分机位;3D 就绪后,四视图变成 3D 渲染的验证基准。四视图本身可以由资产供应链检索现成 + 修正得来,不必都从零生成。

### 5.2 参考素材:独立的 `reference_role` 标签

正交于 Subject 的四类划分(character/portrait/product/scene)。同一张图可能是身份锚点,也可能只是构图/氛围参考;视频参考可能是给运镜镜头感,音频参考是给 timing,不是声音本身。摄入多参考素材时应显式标注这个角色维度。

### 5.3 StylePack:三张可组合查找表(本版核心细化)

**关键认知转变:** StylePack 不是"一段风格描述文本",而是一个**编译器**——把可组合的结构化数据包编译成 provider 适配的 prompt。参考 seedance-prompt 的架构(一个调度器 SKILL.md + 三个词典文件),StylePack 应拆成三个正交、可跨预设复用的原子资产。

#### 5.3.1 根变量(`capture_source`)—— 派生一切的单一来源

每个 StylePack 有一个根字段,描述"这段素材是谁在什么年代用什么设备拍的"。例如:`"2000 年代 DV 家庭录像"` 或 `"35mm 电影感变形宽银幕"`。

这是把"多维度并列描述"改成"单一根变量因果推导"——先锁定来源,光照/运镜/负向词自动派生,一致性问题少一大半。**根变量同时是"追求真实感 vs 追求完美感"的总开关**(见 6.3 适用边界)。

#### 5.3.2 表一:设备包(camera-aesthetics)—— 内容无关的可替换槽位

一个设备包 = `{画面特征, 操作特征, 负面约束}` 三段。参考 seedance 收录的 8 种起步:

`2000 年代 DV / VHS 家用摄像机 / 8mm 胶片 / 手机竖屏 / GoPro / 监控摄像头 / 专业纪录片 / 网络摄像头`

**关键性质:设备包完全独立于内容。** 同一个"社区喂猫"内容,换 DV 包是千禧怀旧,换监控包是悬疑感。因此设备美学在 StylePack 里是一个**可单独替换的槽位**,不焊死在风格描述里。这带来的结构性收益:20 个预设可能只由 8 个设备包 × 1 个共享氛围词典 × 若干内容模板笛卡尔积生成——维护成本从 20 降到个位数,组合数反而爆炸式增长。

#### 5.3.3 表二:氛围词典(atmosphere-dictionary)—— 全局单例,跨所有 StylePack 共享

抽象词 → 可拍摄物理现象的映射表。例:`"温馨"` → `"树影在地面移动 / 晾衣绳织物摇曳 / 陶瓷杯冒热气 / 猫慵懒靠近"`。

**必须是全局单例,不是每个 StylePack 内嵌一份。** 解决的是最要命的问题——抽象词模型无法理解,必须翻译成物理现象。全局共享意味着新加预设直接复用,不重复维护。

#### 5.3.4 表三:provider 默认行为对照表(anti-ai-checklist)—— 每 provider 一份

记录"这个 provider 不加约束时会自动做什么",生成针对性负向词。

**关键认知:负向词不是静态列表,而是对模型默认审美的定向对冲。** 而不同 provider 的默认审美不同——Veo3 默认往电影感走,Kling 默认往丝滑运镜走,本地 Wan 有自己的偏置。所以这张表**每个 provider 一份**,接进已有的"逐镜头 prompt 富化 + provider 适配"步骤。

**自我校正闭环(本版新增):**

```
verdict 观测到某 provider 的 style_distance 偏向
  → 更新该 provider 的默认行为对照表
  → 生成更精准的负向词
  → 下次生成偏置减小
  → verdict 观测到偏置下降
```

这条闭环让 L3 攒的 style_distance 数据反哺 L2 的负向词生成,系统随使用自我校正,不靠人工调参。

#### 5.3.5 非完美事件库 —— 仅 verité 家族启用

四类:摄影师失误、环境介入、主体自然行为、收束方式(seedance 起步 21 个)。时间轴每个镜头塞一个且全片不重复(生成前 lint 校验此项)。AI 视频的"完美感"恰恰是最大破绽,美食/旅行 Vlog 类内容尤其需要这层。**由根变量决定是否启用(见 6.3)。**

#### 5.3.6 版本化

StylePack 版本化,且**校验记录必须钉住生成当时的版本快照**,不能只存"当前版本号引用"——否则 StylePack 升级后历史校验数据全部失真。

#### 5.3.7 创建入口

- 内置预设 fork + 覆盖
- **参考视频/图片 → VLM 拆解调色/光线/运镜/节奏 → 生成 StylePack 草稿 → 用户确认**。给没有自己视觉语言的新用户一个"我要这种感觉"的入口。版权边界:只提取抽象风格语言,不复刻可识别的 IP 角色/画面。

### 5.4 Series(核心资产,战略级)

`Series = {角色组引用, StylePack 引用, 片头片尾模板, 规格锁定(画幅/时长档/分辨率), 集数序列}`。

做第 N 集 = 继承全部 + 只写新剧情。**继承的机制层实现依托 5.6 的七段式字段结构**:第 N 集继承"主要角色 / 视觉风格 / 摄像风格 / 音频"四个字段,只重写"地点 / 时间轴 / 目标"——角色一致性不靠模型记忆,靠字段继承。迁移成本即护城河,是唯一能支撑订阅制的形态。

**翻译配音导出**作为 Series 导出选项:ASR(已有 faster-whisper)→ 翻译(本地 qwen)→ edge-tts 目标语种 → 复用装配管线重烧。全是存量零件组合,一周级工程量。

### 5.5 IP 安全改写 pass

在提示词工程阶段加一道显式改写,处理用户提交内容(含上传照片)里涉及真人/名人/受版权保护角色/品牌 logo 的情况——**改写成安全等价物,而不是简单拦截**。

平台方是担责主体,角色库允许任意照片上传,风险敞口真实存在。

### 5.6 七段式:Director 分镜节点的强制 schema(本版新增)

参考 seedance 的七段式(主要角色 / 地点 / 视觉风格 / 摄像风格 / 时间轴 / 音频 / 目标),**每个镜头节点强制带这七个结构化字段,而不是一段自由文本 prompt**。三大收益:

1. **字段级 lint**:生成前 lint(4.2)可逐字段精确校验,自由文本做不到。
2. **字段级继承**:Series 第 N 集按字段继承(见 5.4),这是角色一致性的机制层实现。
3. **字段级 provider 适配**:不同 provider 语法不同,结构化字段可按 provider 模板重新序列化,而非整段重写。

### 5.7 Subject3D:2D 身份资产 + 3D 视角资产,能力互补共存(v3.1 重写)

**定位(v3.1 修正):** 2D 与 3D **不是"低保真档 / 高保真档"的替代关系**——那个说法暗示它们是同一东西的质量等级、该选高的那个,是**概念错误**。它们是**两种能力不同的资产,并存、互补**:

- **2D 身份资产**(照片/高质量生成图 + identity embedding):**强在身份保真,弱在视角固定**
- **3D 视角资产**(GLB + 多机位渲染帧):**强在视角自由,弱在身份保真(当前档位)**

**两者的强弱在不同维度,根本不该比大小、不该二选一。** 一致性 = 按每个镜头的需求,组合调用两者各自的长处。这才是护城河③(一致性)的正确形态:**不是单靠 3D,是 2D 管"是谁"、3D 管"从哪个角度看",合起来是"从任意角度看都是这张脸"。**

#### 5.7.-1 实测校准:为什么必须共存而非替代(v3.1,来自本地真实跑通数据)

本地 Subject3D(TripoSR 档)真实跑通后的实测:

| 来源 | 身份一致性分 | 视角能力 |
|---|---|---|
| 2D 清晰照片 | **0.77 – 0.84** | 仅固定视角(通常正面) |
| 3D 渲染帧(TripoSR) | **0.61** | **正 / 左 / 右 / 背 多机位** |

**这组数据本身就在说"该共存":** 2D 身份分明显更高(0.77-0.84 > 0.61),但 2D 一张正面照给不了侧/背视角;3D 身份分虽低,却是侧/背视角的**唯一来源**。若二选一(比身份分选 2D),就永久放弃了多机位;若盲目用 3D(信"3D 更高级"),身份反而更差。**结论只能是共存。**

#### 5.7.0 三种镜头级共存消费模式(v3.1 核心)

同一角色的 2D 和 3D 资产并存(施工已实现:subject3d 存 metadata,与 2D 参考图并行不覆盖),消费逻辑按镜头需求组合:

1. **正面/近正面镜头 → 用 2D。** 身份分 0.77-0.84,3D 在这个角度无优势且更差。
2. **侧/背/大角度镜头 → 用 3D 渲染帧。** 2D 在这些角度没有任何信息,3D 哪怕 0.61 也是唯一来源,好过 2D 让模型瞎猜。
3. **最优模式:3D 视角结构 + 2D 身份参考,一起喂 i2v。** 3D 帧当**结构/姿态/视角条件**(定"从这个角度这个姿态看"),2D 当**身份参考**(定"脸是这张脸")。侧脸镜头因此既有正确三维视角(3D 给),又有准确身份(2D 给)——身份分有机会超过 3D 单独的 0.61。**这是共存 1+1>2 的核心:职责分离,各取所长。**

**当前落地状态(如实):** 施工已做到"2D 优先、无 2D 时退回 3D 正面帧",这是模式 1 的保守版;模式 2、3 依赖分镜层有机位/方位角字段才能消费(见 5.7.0.2 未接部分)。数据管道已通,消费逻辑待接。

#### 5.7.-0.5 TripoSR 定位修正(v3.1):共存框架下它可用,不必急着换

v3.0 曾担心"TripoSR 身份分 0.61 太弱,得换 Hunyuan3D"。**共存框架下这个担心大幅缓解:**

- **TripoSR 当"身份来源"不合格(0.61),但当"视角结构来源"够用**——因为身份由旁边的 2D 参考图保证(消费模式 3),3D 只需提供正确的视角/姿态结构。
- 视角结构对 3D 质量的要求,远低于身份保真对 3D 质量的要求。所以一个身份分只有 0.61 的 3D,在"提供视角结构"这个受限职责上可能完全够用。
- **共存不只是更好,还降低了对 3D 质量的要求**,让 TripoSR 这个便宜档(CPU ~172秒/角色)在受限职责下变得可用。是否升级到 Hunyuan3D,取决于"模式 3 合成后身份分能否达标",而非"3D 单独身份分能否追平 2D"——后者是被证伪的错误目标。

**场景资产同理共存:** scene 类 3D 与 character 类平级——场景 3D 提供跨集背景的视角自洽(民国弄堂第 1 集和第 47 集是同一空间的不同投影),而背景的具体质感可由 2D 场景参考补足。场景多机位穿帮最明显(正反打时背景换角度靠 2D 凑不出正确透视),所以场景恰是 3D 视角能力最有用的地方。技术路径:world model → lingbot-map 类前馈重建 → scene 类 3D 资产。

#### 5.7.0.2 机位驱动渲染(v3.0,共存框架下更新;含未接部分如实标注)

分层流水线(4.0)的身份层/场景层,渲染时**以镜头机位为输入参数**,并按共存模式组合 2D/3D:

```
镜头(带机位:正面/侧45°/俯拍/环绕轨迹)
  正面镜头     → 用 2D 身份资产(身份分高)
  大角度镜头   → 3D 渲对应视角帧(结构)+ 2D 身份参考,一起喂 i2v(共存模式3)
  → 合成为该机位的条件帧 → i2v 补动作
```

**正反打不穿帮的根因:** 大角度镜头的视角结构来自同一 3D 资产的不同视角投影(保证空间/姿态自洽),身份来自同一 2D 参考(保证是同一张脸)——两者组合,既不穿帮又保身份。

**⚠️ 当前未接部分(如实标注,施工已明确):** 机位驱动渲染的**数据管道已通**(3D 的正/左/右/背四机位帧已产出并透传到 character_bible),但**消费逻辑未接**——分镜层目前没有机位/方位角字段,镜头不知道自己该用哪个视角的帧。接消费的前置 = **给 shot schema 加机位/方位角字段**。在此之前,实际消费退化为共存模式 1 的保守版(2D 优先,无 2D 时用 3D 正面帧)。这是一个明确的、有前置条件的待办,不是已完成项。

#### 5.7.0.3 3D 化分档(不是全盘 3D,按机位/复用率精准投入)

| 情况 | 用什么 | 理由 |
|---|---|---|
| 主角 + 主场景(高复用、多机位) | 3D 化(Subject3D) | 复用率高摊薄成本,多机位刚需 |
| 配角 + 次要场景(低复用、固定机位) | 四视图 2D 档 | 上 3D 不划算 |
| 一次性路人 / 空镜 | 资产供应链检索现成 / stock | 不值得任何锚点投入 |

判断标准:**要不要多机位 × 复用多少次。** 要多机位且高复用 → 3D;否则 2D 或检索。

#### 5.7.1 进片方式:主路 A + 可选 B + 降级档 C

**主路 A(推荐,复用现有管线):2D 身份 + 3D 视角组合 → i2v 补动态。**
职责切分(v3.1 校准):**3D 提供"从哪个角度看"(视角/姿态/结构),2D 提供"看到的是谁"(身份),视频模型补"怎么动"(动态)。** 注意:v2.2 曾表述为"3D 渲身份帧 100% 一致",实测证明 3D 单独的身份保真弱于 2D(0.61 vs 0.77-0.84),故 3D 不独自承担身份——身份由 2D 参考保证,3D 只保证视角结构。为什么这条最适合 Hevi:完全复用现有生成管线(i2v 接线、逐镜头生成、双变体、verdict 全部不变),3D/2D 都只是在管线最前端接的"参考图来源"(`reference_role`,见 5.2)。风险最低、增量最小。与总纲一致:不自建渲染农场,不绕过视频生成。

**可选 B(降格为 provider,非主干):3D 引擎直渲成片段。**
仅当某个 Series 要纯 3D 动画美学(VTuber/虚拟偶像类)时启用。**降格为 L0 能力矩阵里的一个"渲染 provider"条目**,由路由按 Series 需求选择,而非架构主干。不作为主路的原因:纯 3D 渲染是自建重资产(绑定/动画/物理/光照/渲染农场),工程量高一个数量级,且与"生成能力全部外采"总纲冲突。

**降级档 C(A 的成本档):预渲染多视图当固定参考图。**
A 是"按镜头需求动态渲专属机位帧",C 是"一次性预渲几张标准多视图当固定参考图"。C 更省但丢掉 3D 最强的"按镜头动态出机位"能力。作为 A 的 economy 档:economy 高保真 = 预渲 6 视图;fast 高保真 = 每镜头实时渲专属机位。

**统一判断:主路 A、B 降格为可选 provider、C 作降级档——三者统一在"3D 的职责是提供身份绝对一致的条件输入,不是取代视频生成"。**

#### 5.7.2 统一资产管理层架构

**输入侧(三条摄入管线,全部外采,归一化到统一表示):**

- **图生3D**:用户上传照片/多视图 → 外采图生3D模型(Hunyuan3D/TripoSR 类,作为 L0 一种能力)→ 标准 3D 资产
- **文生3D**:文本/概念 → 外采文生3D → 标准 3D 资产
- **直接导入**:用户上传 GLB/FBX → 校验规格 → 标准 3D 资产

**统一表示:** 三条管线都归一化成一个 `Subject3D`(3D 视角资产),挂在现有 Subject 概念下,与 2D 身份资产并存互补。同一角色同时有 2D 身份资产(embedding/照片)与 3D 视角资产,镜头级按需组合(见 5.7.0),非二选一。

**输出侧(渲染服务):** 给定 `Subject3D + 镜头的机位/光照/姿态需求` → 渲染条件帧 → 进入主路 A 的 i2v。渲染可本地(现有 GPU)或外采。

**关键边界:** 摄入(图生3D/文生3D)全部外采,符合总纲;**自建的只有"统一资产表示 + 渲染编排 + 与 Series/verdict 的集成"——又是编排与资产,不是生成。** 3D 生成能力和视频生成能力一样,是租的、可热替换的;Hevi 的价值仍在管理与编排层。

#### 5.7.3 与 L3 verdict 的互相加强(v3.1 校准:双 ground truth)

共存框架下,verdict 的基准是**两个 ground truth,各管一维**:

- **身份基准 = 2D 身份资产**(实测身份分 0.77-0.84,是当前最可靠的身份真值)。identity_distance 校验以 2D 为基准,而非 3D——这是实测数据(3D 0.61 弱于 2D)的直接推论,修正了 v2.2/v3.0 "3D 当身份 ground truth" 的说法。
- **视角/结构基准 = 3D 视角资产**(几何自洽,是"这个角度该长什么样"的真值)。大角度镜头校验"视角是否正确"以 3D 为基准。

**L2 与 L3 互相加强:2D 提供身份真值、3D 提供视角真值,verdict 分别据此校验身份漂移和视角错误。** 当 3D 质量随图生3D 升级提升(身份分追上或超过 2D)时,身份基准可再评估是否切换——但那要靠数据说话,不靠"3D 更高级"的假设(承接 5.7.-0.5)。

### 5.8 音频资产实体化:声线档案 + 情绪标签(v3.0 新增,让身份从"脸一致"扩到"脸+声一致")

音频不能只是"选个 TTS"。顶级做法是**声线也是挂在角色上的资产**,和身份图对称——挂在 5.0.2 挂载树的 `声线(voice)` 节点上。

- **声线档案**:每个核心角色一份独立声线,跨集锁定。角色 A 永远是这个声音,第 47 集和第 1 集音色一致。实现:edge-tts/vibevoice 选定音色并锁定,或声线克隆(上传样音,支持带呼吸声的情感人声;中文样音训练后可日/英配音用于出海)。
- **情绪标签**:剧本解析(短剧 B0)时,对白自动带情绪标记(如"愤怒:你凭什么这么说"),驱动 TTS 的情感注入,而非平板机械配音。
- **verdict 校验声线**:挂载树遍历时(6.1.0),声线节点校验本镜头音色 vs 声线档案,音色漂移(换 TTS 导致声音变了)会被抓出来。

**意义:身份一致性从"脸一致"扩展到"脸+声一致"——同一个角色第 47 集,是同一张脸配同一个声音。** 这是把视觉资产体系对称地复制到音频侧。

## 6. L3 裁决层:从守门员到记账员

### 6.1 分层校验(替代逐镜头无差别 VLM 审查)

- **Tier 0(结构层,零成本,全量跑)**:ASR 时间戳、时长匹配、字幕对齐率、响度合规。直接拦住"14 镜头压成 6s"这类过压缩 bug。
- **Tier 1(按需视觉,只在 Tier 0 报警的镜头上触发)**:本地 Qwen2.5-VL 抽帧,**遍历一致性挂载树逐节点校验**(见 6.1.0),而非只算两个距离。

原理:大部分结构问题不需要过一次模型就能查出来,VLM 只用在真正需要"看一眼"的边界点上。审片模型本地跑,economy 档也能享受同等质检。

#### 6.1.0 一致性挂载树遍历校验(v3.0 核心升级)

v2.x 的 verdict 只算 `identity_distance + style_distance` 两个数。v3.0 升级为**遍历 5.0.2 的资产挂载树,逐节点独立校验**:

```
校验一个镜头 = 遍历这棵树:
  角色身份一致? (identity_distance vs 2D 身份资产 ground truth,大角度另校视角 vs 3D)
   ├─ 服装一致?  (本镜头服装 vs 该角色当前锁定的那套)
   ├─ 发型一致?
   └─ 声线一致?  (音色 vs 声线档案)
  场景一致?
   ├─ 光照一致?  (时段/色温 vs 场景锁定)
   └─ 道具一致?  (关键道具外观 vs 道具资产)
  任一节点不一致 → 标记该类漂移 → 定向返工(regenerate 带诊断分类)
```

**这是顶级 verdict 与能用 verdict 的分水岭。** 观众吐槽的"女主一套衣服打巴掌关门动作有错"这类穿帮,根因是这些子节点没被当独立实体校验。逐节点校验后,"服装漂移""道具变形"这类以前混在"整体感觉不对"里的问题,变成可定位、可定向返工的具体项。**双 ground truth(5.7.3)让每个节点基准更可靠:身份用 2D 真值,视角用 3D 真值。**

#### 6.1.1 成片交付门清单(v2.3 新增,Tier 0 的成片级扩展)

Tier 0 此前偏"镜头级结构",但成片交付前还有一层"整片级质检"——这些问题**本该被编辑抓到、却常漏给观众**。清单来源是实战出片框架蒸馏的 QA canon(一段视频经 8 轮修复总结),全部是确定性 ffmpeg 检查,零模型成本,作为**成片交付门**在装配完成后、交付前跑一遍:

| 检查项 | 手段 | 拦截的问题 |
|---|---|---|
| **频闪检测** | ffmpeg `blackdetect` + 亮度骤降 | 打斗/闪光/动作镜头的频闪(AI 生成打斗高发,短剧刚需场景) |
| **死空档检测** | ffmpeg `silencedetect`,标记 1.5s+ 句间停顿 | 拖节奏的静音空档 |
| **联络表(contact sheet)** | 每 ~6s 抽一帧拼成一张图 | 一眼扫查画面泄漏/字幕不同步/构图问题——**这是"LLM 不看全帧看压缩表示"的成片版,可直接作为 Tier 1 VLM 或人工审片的输入格式,成本极低** |
| **隐私泄漏** | 检查画面是否漏出浏览器窗口/个人信息 | SaaS 合规项:用户上传素材可能带隐私 |
| **BGM 循环接缝** | 校验 BGM 时长 ≥ 视频时长 | BGM 比片短导致的可闻循环跳变 |

**设计意义:** 联络表这一项尤其关键——它把"整片质检"从"逐帧过 VLM"降成"一张拼图扫一眼",既是确定性 Tier 0 的产物,又是 Tier 1 按需视觉审查的输入载体,一举两得。这份清单把"一个成熟视频编辑交付前会查什么"的隐性知识显性化,不必自己踩坑重新总结。


### 6.2 结果持久化(关键)

每次校验结果写入 `shot_verdict` 表:

```
task_id, shot_id, provider, model_version,
subject_id + 版本快照, stylepack_id + 版本快照,
identity_distance, style_distance,
挂载树逐节点校验结果(服装/发型/声线/光照/道具 各一致性分与通过位),  # v3.0
诊断分类, 重拍次数,
Tier0/Tier1 通过情况, 时间戳
```

四支柱齐全:fingerprint(校验规则版本)、decision_trail(这张表本身)、report(面向客户的连续性报告由此派生)、cost(校验算力)。

**这份数据换来的能力:**

- 把 L0 成本路由从"静态能力表"升级为"活的质量先验"
- Provider 静默降级的自动探测(距离滚动均值台阶式跳变即告警)
- StylePack 调优反馈闭环 + provider 默认行为对照表自我校正(见 5.3.4)
- 系列级慢性漂移检测——单集内每镜头都达标,但跨集维度一致性可能单调爬升式变差,阈值门禁看不见,时间序列一眼看出趋势
- 失败诊断分类的分布统计,指导工程投入优先级
- 为未来蒸馏轻量本地评分器积累训练数据,降低长期审片成本

**这类数据没法补录**——今天不存就是永久丢掉从现在起的信号,持久化应与审片本身同批上线。

**双 ground truth 补充(v3.1 校准):** 身份基准用 **2D 身份资产**(实测 0.77-0.84,当前最可靠身份真值),视角基准用 **3D 视角资产**(几何自洽)。此处修正 v2.2 原表述"identity_distance 优先用 3D 渲染帧当基准"——实测证明 3D 身份分(0.61)弱于 2D,故身份基准应为 2D。3D 承担的是"视角/结构真值",不是"身份真值"。待图生3D 升级后再按数据重评是否切换身份基准(见 5.7.3)。

### 6.3 "缺陷即真实"的适用边界(本版明确)

seedance 方法论通篇讲的是 **verité/纪实/怀旧真实感**这一种美学。"缺陷即真实""把设备物理缺陷一条条喂给模型"只对"要显得像真人随手拍"的内容成立。

但 Hevi 有科普、商务专业、国风水墨等预设,**这些内容恰恰要干净、稳定、完美**。给商务演示视频塞"自动对焦搜索、曝光波动、突然黑屏"是灾难。

**因此:非完美事件库、设备缺陷注入这套逻辑只挂在特定的 StylePack 家族(verité/vlog/怀旧)上,由根变量 `capture_source` 决定开关,不做全局默认。** `capture_source` 是"专业摄影棚"还是"2005 年 DV",直接决定整套缺陷注入逻辑开还是关。这反过来印证了根变量设计的必要性。

### 6.4 面向客户的产出:连续性报告

从 `shot_verdict` 派生"连续性报告"(continuity ledger),随高阶交付一起给用户——零增量计算成本,撑起新的定价层级:普通用户拿 mp4,专业/代理商用户拿 mp4 + 完整制作包(镜头清单、连续性报告、StylePack 引用说明)。

## 7. L4 导演层:唯一出品引擎,Workflow-as-MCP 编排,canvas 升格为 IR

Producer(意图→约束与预算可行性)/ Director(分镜→选角→下发)/ Editor(消费 verdict 评分卡→决定返工)三角色,跑在现有 MCP 四组工具之上,不新建管线。

**外部验证:** 这个 `脚本→生成→审查→重渲染`(script → generate → review → re-render)的闭环,已被独立商业开源项目(Open-AI-UGC)作为核心卖点写进 README,说明这个循环的价值已获市场公认——Hevi 不是在赌一个无人验证的方向。**差异在深度:** 竞品的 review 是浅层的(LLM 看一眼就过),re-render 是整条重来;Hevi 的 review 是分层校验 + 持久化数据资产(L3),re-render 是带诊断分类的镜头级返工(L1)。别人有这两个动作,但没做深——这正是 Hevi 的守土之处。

**关键设计决策:** Director 的输出物就是 canvas 节点图,每个镜头节点带七段式结构化字段(见 5.6)。全自动模式 = 生成图后直接执行;人在环模式 = 用户在画布上改后再执行。canvas 从"一个功能"升格为整个系统的中间表示。

### 7.1 Workflow-as-MCP:导演台调用一切生成能力的统一契约(v2.4 确立)

导演台调用任何生成能力(t2v/i2v/lip-sync/TTS/图生/3D 渲染/装配)的方式,统一为**每个生成步骤都是一个 MCP 工具 / 可热插拔的 workflow**,管线本身**零写死生成逻辑**(参考 Pixelle-Video 的 Workflow-as-MCP Tool 范式)。

- **机制:** Director 产出 canvas 编排图 → 图上每个生成节点解析为一个 MCP 工具调用(底层可以是直连 API、聚合器、或 ComfyUI workflow)。换 provider / 换模型 = 换节点绑定,不改编排逻辑。
- **为什么这是导演台的核心契约而非某出品形式的细节:** 单片、Series、短剧全部通过这套机制编排。它就是"导演台强到能让特化出品只做前置差异"的那个"强"字所在——**下游生成能力一旦全部工具化,任何出品形式都只需产出不同的编排图,不需要碰生成实现。**
- **反自立门户的结构保证:** 因为生成能力只能通过 MCP 工具调用,任何出品形式都无法"绕过管线自己实现生成"——从架构上堵死了短剧那种自立门户的可能。这是把上一轮"短剧不该自立门户"从纪律升级为**结构性不可能**。
- **与 L0 咬合:** ComfyUI workflow(3.5)、聚合器(3.4)、直连 provider 全部可包装为 MCP 工具,能力矩阵与工具注册表是同一张表的两个视图。

### 7.2 出品形式模型:同一引擎的不同输出档位(v2.4 确立)

导演台产出多种成品,它们是**同一引擎按复杂度自动变薄变厚的输出档位**,共享 L0–L3 全部基础设施,区别只在各自特有的前置模块:

| 出品形式 | 特有前置模块 | 共享(全部复用) |
|---|---|---|
| 单片 | 无(最薄档,前置层空转) | 编排范式 / 供给 / Subject / verdict / 成本路由 / 装配 |
| Series | Series 字段继承 | 同上 |
| 头像解说 | 数字人 / lip-sync 路由 | 同上 |
| **短剧/漫剧** | 故事解析层(B0)+ 剧集规划器 + 剧集看板 | 同上 |

- **薄厚自动化:** 单片进来,故事解析/剧集规划空转,走最小路径;短剧进来,点亮那些层。**"不是简单的走旁路,是复杂的点亮更多层。"**
- **无旁路纪律:** 不给任何出品形式(包括最简单的单片)开独立生成/质检/路由旁路。旁路即自立门户,是短剧管道故障的同一模式。
- **新增出品形式的成本:** 加一种出品形式 = 加它的特有前置模块 + 声明它点亮哪些既有层,**不碰引擎本身**。这就是"加强导演台一次,所有出品形式受益"的复利结构。

### 7.3 执行前的计划级自我批判

canvas 草稿生成后,不直接执行,先跑一轮纯推理批判:预算是否超、风格是否和 StylePack 冲突、镜头数是否撑得起目标时长。批判完改一版再执行。零生成成本,拦住"计划本身有问题"这类以前只能靠事后 verdict 发现、但那时钱已花出去的问题。

与 L3 事后裁决前后呼应:**计划前反思,生成后审判,中间才是真金白银。**

### 7.4 创意工具从菜单变成可编排节点

九项创意辅助工具(三视图/分镜网格/多机位/转场/元素编辑等)**不再投入新功能开发**,但也不彻底闲置——它们已是 MCP creative 组里 agent 可调用的形态。真正该做的是让 Director 根据具体任务**自己决定**要不要调用某个工具、怎么串,而非用户在菜单里手动挑。零新增开发,把"人工选择"换成"按需编排"。

## 9. 路线图(依赖关系是硬约束,顺序不可颠倒)

> **护城河优先级标注(v2.1):** 标 🏰 的是三大护城河(路由治理 / verdict 数据资产 / Series 一致性继承)的直接构成项——资源冲突时优先保这些,因为其余项都是可被商品化/外采替代的。

### Phase 1 — 止血与地基(2–3 周)

- L0:余额探针 + Aegis 告警 + 三层预算熔断
- L0:能力矩阵实现为 schema 驱动(可借鉴 SamurAIGPT `schema_data.json` 范式);接入聚合器作兜底供给条目
- L1:i2v 前端暴露、镜头级返工接口(带诊断分类)、生成前 lint
- L1:**readiness 状态机(v3.0,备齐才开机——工程量小,却把生产从抽卡变工序,应尽早)**
- 🏰 L3:`shot_verdict` 表结构与持久化上线(先攒数据,分析可以晚)——这是护城河②的地基,越早存越好,数据无法补录
- 🏰 L2:**资产矿工 + 源资产池起步(v3.0,绿色可商用素材现在就攒,无法补录)**
- 运营:fal 充值;DashScope 明确弃用(LLM 已本地化,wan_cloud 与 wan_local 能力重叠)

### Phase 2 — 质量闭环 + 分层流水线(4–6 周)

- 🏰 L1:**镜头生成分层流水线(v3.0 核心,身份/场景/动作/语音/口型五层串联)+ 分层模型路由表**——这是"用旗舰模型的脸跑本地模型的价"的落地
- 🏰 L2:**五类资产实体 + 挂载树(v3.0,补齐服装/道具/声线三类,分层挂载)**
- 🏰 L2:Subject 四视图标准包(v3.0,2D 多机位最低标准)+ identity embedding;IP 安全改写 pass
- 🏰 L3:分层校验 + **一致性挂载树遍历校验(v3.0,逐节点校验替代两个距离)**
- 🏰 L3:成片交付门清单(频闪/死空档/联络表/隐私/BGM接缝——纯 ffmpeg,零模型成本)
- L1:口型独立层(路由到专用音画同步模型)
- 音频:**声线档案 + 情绪标签(v3.0,音频资产实体化)**
- L0:成本感知路由 v1(镜头级 + 分层 provider 选择);stock_footage 对位校验;资产供应链检索匹配
- L1:跨 provider 调色统一 + BGM 节拍对齐 + 多轨同步对齐范式(根治过压缩 bug)

### Phase 3 — 产品跃迁(6–8 周)

- **L2:StylePack 重构为三张查找表(设备包 × 全局氛围词典 × provider 默认行为对照表)+ 根变量 + 七段式 schema + 非完美事件库**
- 🏰 L2:Series 资产化 + StylePack 版本化 + 参考视频创建入口(护城河③)
- L2:翻译配音导出
- L3:连续性报告作为付费分层;provider 默认行为对照表自我校正闭环
- L1:长视频切片分发(外采,作 Series 下游导出选项)
- L4:Director Agent v1(先做 Producer 单角色,Editor 依赖 verdict 成熟后再上)
- 🏰 **L2:Subject3D 探路 —— 本地已跑通(✅ 部分完成)** —— 本地 TripoSR 生成模块已跑通验证:GLB + 正/左/右/背 4 机位渲染帧,~172秒/角色(CPU),存 metadata 与 2D 参考图并存不覆盖,短剧派发后台 fire-and-forget 生成。**实测发现:3D 身份分 0.61 < 2D 0.77-0.84,据此确立 2D+3D 互补共存(见 5.7)。剩余:机位消费逻辑未接(分镜层缺机位字段)、TripoSR 是否够用待共存模式3验证。**

### Phase 4 — 导演化 + 3D 视角资产深化(依赖 Phase 2/3 完成)

- L4:计划级自我批判、创意工具动态编排接入
- 🏰 L0:成本路由 v2(用 verdict 历史数据驱动)——护城河①的兑现,依赖 Phase 1 起持续积累的 shot_verdict 数据
- 🏰 **L2:Subject3D 深化(接共存消费 + 提质)** —— (1)给 shot schema 加机位/方位角字段,接通机位消费逻辑(当前只有数据管道);(2)实现共存模式3(3D视角结构 + 2D身份参考一起喂 i2v),验证合成后身份分能否超 0.61;(3)据验证结果决定是否从 TripoSR 升级到更强图生3D(Hunyuan3D 类);(4)推理从 CPU 迁 GPU/云(172秒/角色是验证期数字非生产期);(5)角色 + 场景 3D 平级。B 路线(3D直渲 provider)按需在有 VTuber 类需求时再接。
- 观察项:world model → 3D 场景重建(lingbot-map 类),补强 scene 类 3D 视角资产,待可商用 license + 硬件路径成熟触发。

**硬约束重申:** 没有 L3 持久化数据,L0 路由 v2 无从谈起;没有 L4 计划级反思,自动化越高烧穿预算风险越大;StylePack 三张查找表是 Series 继承和 lint 的共同前置;**分层流水线是分层模型路由、机位驱动渲染、口型独立层的共同前置,应作为 Phase 2 的主干优先落**;**五类挂载树是一致性遍历校验的前置,两者同批**。3D 视角资产依赖 i2v 管线(Phase 1)与 Series(Phase 3);本地探路已跑通(见 Phase 3),深化排 Phase 4;主路 A(2D身份+3D视角组合)复用现有管线,不阻塞其他工作。

**v2.1 补充纪律:** 三大护城河的构成项(🏰)在任何资源取舍中优先级最高。凡是可被聚合器/服务化替代的能力(接入、切片、单点生成、3D 生成),一律外采,不占自研资源——省下的深度全部投给别人做不深的那三样(路由治理、verdict 数据资产、一致性继承,其中一致性继承的终极形态即 3D 资产 + 五类挂载树)。

## 附录 C:护城河模型(一页速览,v2.2 更新)

```
已被商品化(不是护城河,外采即可):
  ├── 单镜头生成能力        → 租(fal/直连)
  ├── 多 provider 接入      → 聚合器一行接入(muapi 类)
  ├── lip-sync / 运镜 / 特效 → 模型层吃掉
  ├── 长视频切片分发        → 服务化(AI Clipping 类)
  └── 3D 模型生成能力       → 外采(图生3D/文生3D,Hunyuan3D/TripoSR 类)

Hevi 真正的护城河(接入层之上,竞品未做深):
  ├── ① 路由治理    —— verdict 质量数据驱动的镜头级 provider 选择(L0 v2 + 5.3.4 闭环)
  ├── ② verdict 数据资产 —— 分层校验 + 持久化 + 身份 embedding(L3,别人的 review 是浅层的)
  └── ③ 一致性继承 —— 两个层次:
         ├── 2D:七段式字段继承的"第 N 集"量产机制(L2,单片赛道无此物)
         └── 3D:Subject3D 统一资产层,身份"结构上不可能不一致"(L2,一致性的终极形态)
              注:3D 模型"生成"外采,但"统一资产表示 + 渲染编排 + 与Series/verdict集成"自建

三者的共同点:都不是"能不能做一次",而是"能不能稳定重复地做,并随使用越做越好"。
这是聚合器和单片工具结构上给不了的东西。

关于 3D 的判断口诀:3D 生成是租的,3D 资产管理是护城河。
Hevi 不做"造 3D 模型"这件被商品化的事,做的是"让 3D 资产稳定服务于第 N 集身份一致性"这件没人做深的事。
```

## 附录 D:v3.0 顶配九项密度升级(一页速览)

```
v2.x 确立骨架(五层 + 导演台唯一出品),v3.0 填密度到顶级:

生成侧(把"一次生成"拆成工序):
  ① 分层流水线      身份→场景→动作→语音→口型,每层独立选模型
  ④ readiness 状态机 备齐才开机(draft→ready→gen→verdict→done),抽卡→工序
  ⑦ 口型独立层      路由到专用音画同步模型,不指望动作层附带

资产侧(把"锁向量"升成"管资产体系"):
  ② 资产供应链      矿工采集→检索最接近→修正锁定;检索优先,生成兜底
  ⑤ 五类实体+挂载树  角色/场景/道具/服装/声线,分层挂载成树
  ⑥ 四视图标准包    正/侧/背/特写,2D 多机位最低标准,3D 前置
  ⑧ 声线档案        音频资产实体化,身份从"脸一致"扩到"脸+声一致"
  ③ 场景3D+机位     场景3D与角色平级;机位是2D/3D分水岭,驱动渲染

裁决侧(把"锁两个数"升成"遍历一棵树"):
  ⑨ 挂载树遍历校验   逐节点校验衣/发/声/光/道具,而非只算 identity+style

顶级 vs 能用 的分水岭(实证结论):
  能用的:锁脸 + 锁风格,一次生成,交付即结束
  顶级的:锁脸+衣+发+声+道具+光(遍历树),分层流水线,备齐才开机,检索优先
          — 一致性/资产化/多视图已是行业 table stakes,顶级差在"密度"
```

一句话:**v3.0 不加新赛道,只把每一层做到工业顶配的密度——生成是工序不是抽卡,资产是体系不是向量,裁决是遍历树不是两个数。**
